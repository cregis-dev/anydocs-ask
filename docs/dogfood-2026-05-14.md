## Findings（按严重度倒序）

### F1【bug, P1, fixed by #18】`/v1/ask` LLM 抛错时 run 不落 jsonl

**症状**：10 条 console persist 查询里 2 条返回 30s 超时；console 日志显示 `AnthropicLLM: gateway returned non-object response`；`runs/2026-W20.jsonl` 只新增 8 行（应为 10 行），失败的 2 条**完全消失**。

**根因**：[src/server/app.ts:107](src/server/app.ts#L107) `askWithTrace` 调用未被 try/catch 包住；LLM 失败抛 Error → Hono 自动 500 → `appendRun` 永不执行。注释 line 112-113 明确说「analyze needs visibility into errors / clarifies / answers alike」，但实现路径只覆盖了 `result.type === 'error'` 的「expected error」，没覆盖 LLM 抛 throw 的「unexpected error」。

**影响**：
- runs.jsonl 不再是 `/v1/ask` 流量的完整审计日志（违反 ARCH §16.4）
- analyze D1/D2 看不到 LLM 间歇性失败的信号 → 作者无法定位「gateway 不稳定」这类系统性问题
- 用户看到 500 Internal Server Error，没有 actionable hint

**建议修法**：在 line 107 外层加 `try / catch (err)`，把 LLM throw 合成 `{ type: 'error', code: 'llm_failed', message: err.message }`，落 run，返回 503（与已有 `llm_unavailable` 一致）。golden generate 流早就走通这个模式（PR #14 `emitError`），现在 ask 路径需要补齐。

### F2【bug, P1, fixed by #19】Golden generate `--limit N` 偏置：lang 顺序导致小 limit 只产出非默认语言

**症状**：项目 `defaultLanguage: "zh"`、`languages: ["zh", "en"]`，跑 `+from structure --limit 30` 得到 30/30 候选**全是英文**（"What is X?" / "How do I use X?"）。

**根因**：[src/golden/generator.ts:80-91](src/golden/generator.ts#L80-L91) `langs = Array.from(project.navigationsByLang.keys())` 用 Map 插入顺序，然后 `out.slice(0, limit)` 在所有语言展开完之后切。en 比 zh 先入 Map（推测来自 anydocs 主仓 loader 的字典序）→ 38 页 × 5 模板 = 190 个 en 候选填满前 30 名。

**影响**：
- zh-default 项目的作者「+from structure」拿到 0 个母语 case
- 配合 F3（LLM rewrite 同时失败）→ 评测集对项目主语言完全不可用
- 误导作者以为模板候选质量差，实际是 lang 路由问题

**建议修法**：在 generator 接受 `defaultLanguage` 提示，或按 `project.config.defaultLanguage` 优先排序；至少把 `--limit` 改成**按 lang 等比分配**（30 limit + 2 langs → 15+15）而不是先 en 再 en 再 zh。也可以在 console UI 加 lang 选择器。

### F3【可观测性, P2】LLM 失败错误信息留 `undefined`，不可定位

**症状**：3 次 LLM 失败均报：
```
AnthropicLLM: gateway returned non-object response (model=deepseek-v4-pro): undefined
```
`undefined` 后没有原始网关响应、HTTP 状态码、或截断 body。作者无法判断是 401/网络中断/JSON 解析失败/还是空对象。

**建议修法**：[src/llm/anthropic.ts:80](src/llm/anthropic.ts#L80) 附加 `JSON.stringify(rawResponse).slice(0, 200)` 或显式的 `(status: 502, body: "...")`。

### F4【UX, P2】Citation 渲染：同页两 chunk 视觉重复

**症状**：首问 "如何在 CLI 中初始化一个 anydocs 项目？" 召回 2 条 citation，都是 page_id=`installation`、title=`安装`、breadcrumb 完全一致；只有 `in_page_path`（`创建一个文档项目/p[1]` vs `p[1]`）不同。UI 渲染时 title 24px 加粗，`in_page_path` 在 meta 行小灰字 → 视觉上看起来是「同一条 citation 出现两次」。

**建议修法**：[src/console/pages/project.ts:1207-1214](src/console/pages/project.ts#L1207-L1214) 把 `in_page_path` 拼到 title 后或提到 title 同级；或同 page 多 chunk 时显示 `安装 · §创建一个文档项目` / `安装 · 引言`。

### F5【copy, P3】项目页 Ask 卡停止态 heading 仍中文，与英文 body 不一致

**症状**：PR #13 把 next-action banner 改写为 plain English；但 Ask 卡停止态保留 `<h2>项目未启动</h2>`，下一行 `<p>` 已是 `Start project to ask questions, run eval, and record traffic. First boot loads the embedding model and indexes pages — usually 5–30s.` 同一卡片中英混排。

**建议修法**：[src/console/pages/project.ts](src/console/pages/project.ts) 搜 `项目未启动` 改成 `Project not running` 或反之统一回中文（取决于 IA cleanup 的最终方向）。

### F6【performance 观察, P3】analyze D1 在小项目上 9/9 命中召回失败

**症状**：10 条真实查询（其中 8 条成功 answer，2 条 clarify）跑 `analyze runs --since 1d --include_console`，D1 报 9/9 都低置信度。其中 5 条主观判断答得不错（`installation` / `deploy` / `ai-artifacts` 命中合理），不该被全部计成 recall failure。

**根因猜测**：`confidence = top1.final_score / sum(top-5.final_score)` 在 76 页量级、5 路融合较均匀时天然 ~0.2-0.4，低于默认 floor。

**建议**：先记录 — 等积累更多项目数据再决定 floor 是否需要校准（PRD §16.6 已留口子）。**不在此版本动**。

### O1【次级观察】LLM 失败时单次调用 timeout 偏长

**现象**：Golden generate 失败：`retry 1/2 after 2000ms ... retry 2/2 after 4000ms ... FAIL in 97811ms`。除去 2s+4s backoff，每次 LLM 调用本身要 ~30s 才 timeout。30 模板 + 单 batch（30 items） + 3 次尝试 ≈ 96s。

**建议**：加可配置 `singleCallTimeoutMs`（默认 20s 或更短），让 fallback 路径更快触发。当前 console 已经走自动降级，但 97s 的等待对一键按钮的「秒级反馈」感受很伤。先记录 — 优先级低于 F1/F2/F3。
