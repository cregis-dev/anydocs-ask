## 发现的瑕疵

### F7【data, P2】β feedback row 的 `generated` 列总是空字符串

**症状**：drawer ANSWER 段显示 "(no answer body)"。即使 ask happy-path 返回完整 markdown，β 提交后 `feedback.generated` 列 length=0。

**根因**：[src/server/app.ts:318](src/server/app.ts#L318) 的 INSERT 语句对 `generated` 列只读 `obj.generated`（请求体字段）。Reader / curl 客户端都不发 `generated`，所以入库总是空字符串。同时 `answers.payload` JSON 里其实有完整 `answer_md`——只是 β handler 没去取。

**修法**：与 PR [#48](https://github.com/cregis-dev/anydocs-ask/pull/48) 同 pattern（question 回填），从 `answers.payload` 解出 `answer_md` 作为 generated 默认值，请求体显式传 generated 时优先。

**为什么 P2 而非 P0**：β feedback 数据本身可用（rating + correction 都正确入库），drawer 上 "no answer body" 不影响 KPI / 分组 / 校验。但 Studio 抽屉的"看到上下文做后处理"场景被破坏。

### F8【UI, P2】Feedback drawer 错显 "no citations on this answer"

**症状**：T2 答案明明返回 cit=2，drawer CITATIONS 段显示 "no citations on this answer"。

**根因**：[src/console/feedback-state.ts:663-680](src/console/feedback-state.ts#L663-L680) 的 `parseRunCitations` 期望每个 item 有 `.page` 字段：

```ts
.filter((c): c is { page: string; ... } => typeof c?.page === 'string',
```

但 `feedback.retrieved` 列存的是 `Citation[]`（query/types.ts 的 shape，字段叫 `.page_id`），不是 `RunCitation[]`（runs/types.ts 的 shape，字段叫 `.page`）。所有 item 都被 filter 掉，输出空数组。

**修法**：让 parser 同时识别两种字段（`page` 优先，`page_id` 兜底），或在 `/v1/ask/feedback` 入库时把 `Citation[]` 标准化成 `RunCitation[]` shape 再 JSON.stringify。后者更干净但需要 schema 版本对齐。

**verified**：DB `feedback.retrieved` 列 length=960/2346 字节有完整数据；API 输出 `citations_count: 0`；典型的字段名漂移 bug。

### F9【UX, P3】Reader HISTORY 抽屉不实时反映新建 session ✅ 已修（2026-05-24）

**症状**：浏览器开 `/ask`，问答后展开左侧 HISTORY 抽屉显示 "No conversations yet"，但 localStorage 实际已有 active session + 完整 snapshot。

**根因**：`openHistory()` 已经每次打开都重读 localStorage——只是 user 在第一次问答前先打开抽屉的场景没人触发重新渲染。`upsertCurrent` 写入 localStorage 后没通知开着的抽屉。

**修法**：`upsertCurrent` 在写完 localStorage 后检查 `histDrawer.hidden`，若处于打开状态立刻 `renderHistory()`。一行 hook，不引入 storage event。

**真机验**：单测覆盖 inline JS 渲染断言。浏览器侧 dogfood 留下次 session。
